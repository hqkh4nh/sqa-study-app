Software Testing Methods & Techniques

Two approaches: static testing checks defects without executing code; dynamic testing validates functionality by executing it. Covers review, static analysis, the dynamic process, and how the two compare.

01 — The Two Approaches

Software Testing Approaches

Testing branches into two approaches — both target defects, but differ in whether the code is run:

Static testing

A type of software testing method used to check defects without executing code.

Dynamic testing

The type of testing that validates the functionality of an application by executing the code.

02 — Static Testing

1. Static Testing

  • Static testing = a software testing method used to check defects without executing code.
  • Primary goal: to catch defects in the documentation phase — prevents errors at later stages, so it is cost-effective.

Two main technique types are used in static testing:

Review

A process to find potential defects. Forms: Informal, Walkthrough, Peer review, Inspection.

Static Analysis

Examines source code and models without execution. Types: Data flow, Control flow, Cyclomatic complexity.

03 — Review

1.1 Review

Definition

Review is a process or technique that is performed to find the potential defects in the design of the software.

Performed to identify and correct errors in documents such as requirements, design, implementation, test cases, and maintenance.

Types of Review — Levels of Formality

Review types ordered by level of formality, lowest (top) to highest (bottom):

Review typeLevel of formality
InformalLow
Walkthrough↓
Technical review↓
InspectionHigh

Informal Review

  • Applied in the early stages of the life cycle of the document.
  • Conducted between two persons; the main purpose is for "discussion".
  • Not based on a procedure and not documented.

Walkthrough

  • A formal review process, usually in the meeting room; led by the authors.
  • The author guides participants through the document by his or her thought process to achieve a common understanding and to gather feedback.
  • Goal: finding defects and helping team members understand the document's content.

Technical Review (Peer Review)

  • Technical reviews are documented; a discussion meeting focused on the technical content of the document.
  • Defects are found by the experts (such as architects, designers, key users) who focus on the content.
  • Goal: to ensure the value of technical concepts are used correctly.

Inspection

  • The most formal review technique; led by trained moderators.
  • Reviews business requirements, functional requirements, system design, code, and testing activities.
  • Has defined entry and exit criteria for the product under review.
  • Goal: find defects as early as possible; improve the quality of the document under inspection; reports/documents are maintained and kept for future records.

Inspection Review Process (Formal Review)

A repeating cycle of six stages:

1. Planning2. Kick-off3. Preparation
4. Review Meeting5. Rework6. Follow-up

04 — Static Analysis

1.2 Static Analysis

Static analysis is performed on requirements, design or code without actually executing the software (or before the code is run). Goal: find defects in software source code and software models.

Typical Defects Discovered by Static Analysis Tools

  • Unused variables
  • Dead code (or unreachable code)
  • Infinite loops
  • Variable with undefined value
  • Security vulnerabilities
  • Syntax violations of code and software models

Benefits of Static Analysis

  • Early detection of defects prior to test execution.
  • Early warning about suspicious aspects of the code or design, by the calculation of metrics such as a high-complexity measure.
  • Identification of defects not easily found by dynamic testing.
  • Improved maintainability of code and design.
  • Prevention of defects.

Three Types of Static Analysis

TypeDescription
Data FlowRelated to the stream processing.
Control FlowBasically how the statements or instructions are executed.
Cyclomatic ComplexityThe measurement of the complexity of the program, basically related to the number of independent paths in the control flow graph of the program.

05 — Dynamic Testing

2. Dynamic Testing

Definition

Dynamic testing validates the functionality of an application by executing the code — giving input values and validating the expected output against the actual output.

Divided into two categories:

  • White box testing
  • Black box testing
Example: Testing login functionality where the requirement is that the Username is restricted to Alphanumeric — supplying inputs and checking the actual output against the expected output.

06 — The Process

Dynamic Testing Process

Flows through five steps:

StepActivityDetails
1Test case designDesign test cases based on the requirements; identify features to be tested; derive the test conditions; derive the coverage items.
2Test environment setupInstall the build and manage the test machines; make sure the testing environment is always comparable to the production environment.
3Test case executionTest cases are actually executed.
4Test analysis & evaluationAnalyze and evaluate the findings by comparing the actual results with the expected results.
5Bug reportingIf the actual results and expected results are not the same, the test case is marked as Fail and a bug should be logged.

07 — The Comparison

Static Testing vs Dynamic Testing

Static testing

Examines work products without executing the code; finds defects; aims at finding defects early in the life cycle; rework cost is relatively low; includes the verification process; assesses the documentation and code.

Dynamic testing

Involves the execution of the code; finds the failures in the system; carried out during and after the development phase; rework cost is relatively high; includes the validation process; finds the bugs in the software system.

08 — Takeaways

Summary

Static testing finds defects without running the code; dynamic testing finds failures by running it — together they cover the life cycle.
  • Static testing uses review (informal, walkthrough, technical review, inspection) and static analysis (data flow, control flow, cyclomatic complexity) — catches defects early and cheaply via verification.
  • Dynamic testing executes the code through a five-step process to validate behavior and find failures.
