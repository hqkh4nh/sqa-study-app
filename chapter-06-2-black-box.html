Black-Box Testing

Black-box testing checks what the software does, not how — against requirements and specifications, without knowledge of the internal code. This chapter reviews the six common techniques: equivalence class, boundary value, combinatorial/pairwise, state transition, decision table, and use case testing.

01 — The Method

What is Black-Box Testing?

Definition

Black box testing is a software testing method in which the functionalities of software applications are tested without having knowledge of internal code structure, implementation details and internal paths.

Also known as specification-based testing and behavior-based testing. The tester focuses on what the software does, not how — testing is based on software requirements and specifications. Inputs go into the "black box" and outputs come out, without knowledge of the internal workings.

Black-box testing consists of functional testing and non-functional testing.

02 — Overview

Black-Box Testing Techniques

The common techniques of black box testing are:

  1. Equivalence Class Testing
  2. Boundary Value Testing
  3. Combinatorial / Pairwise / All-Pairs Testing
  4. State Transition Testing
  5. Decision Tables Testing
  6. Use Case Testing

03 — Technique 1

Equivalence Class Testing

Also known as Equivalence Partitioning.

Idea

Divide or partition the input domain of a program into classes of data, and derive test cases based on these partitions.

Goal: to reduce the total number of test cases to a finite set of testable test cases, still covering maximum requirements.

The system handles all the test input variations within a partition in the same way:

  • If one input condition in a partition passes, then all other input conditions within the partition will pass as well;
  • If one input condition fails, then all other input conditions within the partition will fail as well.

Procedure: first-level partitioning separates Valid vs. Invalid test cases; then valid and invalid cases are partitioned into equivalence classes; finally a test case is created for at least one value randomly selected from each equivalence class.

Weak vs. Strong Equivalence Class Test Cases

Consider a program with 2 input variables x and y, where x lies in 3 possible domains (a ≤ x < b, b ≤ x < c, c ≤ x ≤ d) and y in 2 possible domains (e ≤ y < f, f ≤ y ≤ g).

Weak equivalence class test cases

A small set that covers each variable's classes but not all combinations — one test per class per variable.

Strong equivalence class test cases

The full Cartesian product of equivalence classes — one test for every combination of an x-class and a y-class (the larger set).

The Triangle Problem

A program takes 3 integer inputs (side lengths) and determines whether they form an equilateral, scalene, or isosceles triangle, or do not represent a triangle. For a, b, c to be triangle sides: a + b > c, a + c > b, and b + c > a.

Input test data is defined via equivalence classes through the 5 output groups: not a triangle; isosceles; scalene; equilateral; and error inputs. Test conditions:

ClassCondition
Error Inputa <= 0 || b <= 0 || c <= 0
Not a Trianglea + b <= c || b + c <= a || a + c <= b
Equilaterala == b && b == c
Scalenea != b && b != c && a != c
Isosceles(a == b && b != c) || (b == c && c != a) || (a == c && a != b)

Simpler example — a field accepting Age 18–56 yields 3 equivalence classes and 3 test cases: Valid Class 18–56, Invalid Class 1 (<=17), Invalid Class 2 (>=57).

04 — Technique 2

Boundary Value Testing

Boundary / Limits Testing tests values, sizes or quantities near the design limits (value limits, length limits, volume limits, null strings vs empty strings). Errors tend to occur near the extreme values of inputs. Robustness asks: how does the software react when boundaries are exceeded?

Basic idea (BVA)

Use input variable values at their minimum, just above the minimum, at a nominal value, just below the maximum, and at the maximum — for a variable x where a ≤ x ≤ b, the five values are x(min), x(min+), x(nom), x(max-), x(max).

Single Fault Assumption

The Single Fault assumption from reliability states: "Failures are rarely the result of the simultaneous occurrence of two (or more) faults."

For 2 variables (a ≤ x₁ ≤ b and c ≤ x₂ ≤ d), test cases form a plus/cross shape: one variable takes its extreme values while the other is held at nominal, plus a central nominal point — two variables rarely assume their extreme values simultaneously.

Extensions of Boundary Testing

TechniqueBehavior
Boundary value testingStay within the limits.
Robustness testingExceed the limits — include values below the minimum and above the maximum.
Worst-case testingLike boundary value testing, but discard the Single Fault Assumption — allow input values to simultaneously approach their boundaries.
Robust worst-case testingLike robustness testing, but discard the Single Fault Assumption (the largest set of combinations).

Robustness testing is negative testing — how does the program handle errors in the input? It stresses input boundaries, checks acceptable responses to invalid inputs, and leads to exploratory testing. It may not be possible with some strongly-typed languages, GUIs with fixed palette values, or drop-down lists.

Example — Triangle Program: side lengths are integers from 1 to 200. BVA requires 5 values per variable (min, min+, nominal, max-, max), so boundary testing for 3 variables requires 5 × 3 = 15 test cases (values 1, 2, 100, 199, 200).

05 — Technique 3

Combinatorial / Pairwise / All-Pairs Testing

Definition

Pairwise testing (also known as All-Pairs or Combinatorial testing) is a black-box test design technique in which test cases are designed to execute all possible discrete combinations of each pair of input parameters.

Pairwise suites cover all combinations of two parameters, so they are much smaller than exhaustive ones yet still effective at finding defects. Example: a GUI with List Box (10 values), Check Box (2), Radio Button (2) and Text Box (1–100) gives an exhaustive Cartesian count of 10·2·2·100 = 4000 test cases; reducing each parameter (e.g. Text Box → Valid/Invalid/Alpha-Special) and applying all-pairs brings this down to just 6 test cases.

PICT and Other Automation Tools

Pairwise tools automate test case design by generating a compact set of parameter value choices. The key tool is PICT — "Pairwise Independent Combinatorial Testing", provided by Microsoft. Others: IBM FoCuS, NIST ACTS, Pairwise by Inductive AS, and VPTag.

Modeling Steps

To model a system for PICT, list parameters affecting a single common output (e.g. font, style, effects, color, size). Then refine the model:

  • Abstraction — simplify the model by grouping parameters, e.g. style == bold and italic, effects == underline and strikethrough.
  • Equivalence-partition ranges of values — avoid hard-coding variables in a range; map continuous ranges (e.g. size 1–1637.5) onto symbolic values (small, nominal, medium, large, larger, largest).
  • Aliasing — alias (equivalence) similar variables, e.g. font: arial | tahoma, brushScript, monotypeCorsive, when two fonts are similar but both should be tested.
  • Constraints — tweak the model, not the output; constrain invalid combinations with conditional rules (e.g. if [Font] = "BrushScript" then [Style] in {"Italic","Bold/Italic"}).
  • Weighting — increases the probability of a variable's use, but does not guarantee increased use.
  • Increase order of combinations — use the PICT /o:n switch to go from 2-wise to 3-wise for greater depth; the /r:n switch randomizes output to increase breadth.

Always verify the output for mutually exclusive (invalid) combinations, and use only one negative value per test, separating negative and positive tests.

06 — Technique 4

State Transition Testing (STT)

Definition

State Transition testing is the software testing technique in which changes in input conditions cause state changes. It is helpful where you need to test different system transitions.

Tests are designed to execute valid and invalid state transitions:

  1. Step 1: Understand the various states and transitions and mark each valid and invalid state.
  2. Step 2: Define a sequence of events that leads to an allowed test ending state.
  3. Step 3: Each visited state and traversed transition should be noted down.

Steps 2 and 3 repeat until all states are visited and all transitions traversed. For good coverage, actual input and output values must be generated.

When to use STT

Testing for a finite set of input values; testing a sequence of events that occur in the application; when the system has a dependency on past events/values.

When NOT to use STT

When testing is not done for sequential input combinations; when testing different functionalities like exploratory testing.

Steps of STT

  • Identify the States / Events / Actions;
  • Identify the Transitions between them;
  • Create the diagram;
  • Create the State Transition Table;
  • Create the Test Scenario;
  • Create the set of test cases.

Components of an STT Diagram

Two components: States/Events/Actions (a named oval node) and Transitions (a named arrow between states).

For the ATM PIN example (the account locks after three invalid attempts), the state transition table is:

07 — Technique 5

Decision Table Testing

Definition

Decision table testing is a black-box test design technique to determine the test scenarios for complex business logic. It helps testers search the effects of combinations of different inputs.

Equivalence Partitioning and BVA apply only to specific conditions or inputs. A decision table identifies conditions/inputs and expected outcomes/actions in a table, then derives tests from the table rules. It is an effective way to model programmatic logic: a table specifying the actions to take when certain conditions arise.

Limited Entry vs. Extended Entry

Limited entry decision table

Each condition value is binary (T/F).

Extended entry decision table

Conditions can have values (more abstract condition entries).

Structure of a Decision Table

  • The condition stub is a list of names of conditions.
  • The action stub names the actions the routine will take or initiate if the rule is satisfied.
  • The condition entry is the combination of conditions.
  • The action entry are the actions that take place under combinations of conditions.
  • Each column is a rule; each rule is a test case.
A redundant table has identical rules with identical action entries (no ambiguity, just redundancy). An inconsistent table has identical condition entries but different action entries — implying non-determinism, so you can't tell which rule to apply.

Triangle Problem (Limited Entry)

A limited entry table for the triangle problem builds rules from binary conditions (A+B≤C, B+C≤A, A+C≤B, A=B, B=C, A=C) producing actions Invalid, Equilateral, Scalene, Isosceles and N/A. Impossible conditions are trimmed to prevent false positives, leaving eight test cases that test only the software's logic (e.g. Rule 4 → (60,60,60) Equilateral; Rule 11 → (30,40,50) Scalene).

Extended Entry — Next Date

The Next Date program takes month, day and year and computes the next date; a limited entry table would require approximately 256 rules. Using equivalence partitions (Months M1–M4, Days D1–D5, Years Y1–Y2) and pruning N/A rules, the condition entries become more abstract and the table shrinks dramatically.

Another Example — Disc Space Access

For a system checking valid username, valid password, and account in credit, the eight T/F combinations are rationalised using a hyphen for "don't care"; expected output conditions are determined per column, and each column (tagged A, B, C, D) becomes at least one test case:

08 — Technique 6

Use Case Testing

Definition

Use case testing is a black-box testing technique that helps identify test cases that cover the entire system.

It is helpful for deriving test cases for integration testing, system testing and acceptance testing. Each use case describes the interaction between the actor and the system to achieve a specific task.

A use case specifies a Use case Name, Actors, and Pre-Conditions, then a Main Scenario (normal flow of actor/system steps) plus Extensions (alternative or error flows). Test cases are derived from the normal flow and each extension — e.g. the "Show Student Marks" case derives a normal-flow test and an invalid-ID test from its extension steps (3a invalid ID shows an error; 3b invalid ID entered 4 times closes the application).

09 — Takeaways

Summary

Black-box testing checks behavior against specifications — choose the technique that fits the input domain, the combinations, the sequences, or the logic under test.

Equivalence class testing partitions the input domain; boundary value testing probes the extremes (with robustness, worst-case and robust worst-case extensions governed by the Single Fault Assumption); combinatorial/pairwise testing (via PICT) shrinks an exhaustive set while covering all pairs; state transition testing covers event-driven state changes; decision tables model complex logic as rules; and use case testing derives end-to-end tests from actor–system interactions.
