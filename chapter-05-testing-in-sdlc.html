Software Testing in the SDLC

Testing's place in the development life cycle: the V-Model, verification vs validation, QA vs QC. The four test levels — component, integration, system, acceptance — and the four test types: functional, non-functional, structural, and testing related to changes.

01 — Role in the SDLC

The Role of Software Testing in the SDLC

A development life cycle captures the initial requirements from the customer, expands them into the detail required for code production, then writes and tests the product ready for release. Two common families of models:

  • Sequential development models — Waterfall, V-model, …
  • Iterative and incremental development models — Scrum, Spiral, Agile, Kanban, Rational Unified Process, …

02 — The V-Model

The V-Model

The two branches of the V symbolise the relationship between development and testing. The left branch represents the development process; the right branch defines a test level for each specification and construction level. Each left level links horizontally to its matching right test level (Verification & Validation); implementation flows down the left, integration up the right (Implementation & Integration).

Left branch — the development process

  • Requirements definition — capturing of user needs.
  • Functional specification — definition of functions required to meet user needs.
  • Technical system design — technical design of functions identified in the functional specification.
  • Component specification — detailed design of each module or unit to be built to meet required functionality.
  • Programming — use a programming language to build the specified component (module, unit, class…).

Right branch — a test level for each level

Each test level is linked horizontally to its matching development level:

Development level (left)Test level (right)What it verifies
Component specificationComponent testVerifies whether each software component correctly fulfills its specification.
Technical system designIntegration testChecks if groups of components interact in the way specified by the technical system design.
Functional system designSystem testVerifies whether the system as a whole meets the specified requirements.
Requirements definitionAcceptance testChecks if the system meets the customer requirements (as in the contract) and/or user needs and expectations.

Characteristics of the V-Model

  • For every development activity, there is a corresponding testing activity.
  • Each test level has test objectives specific to that level.
  • Test analysis and design for a given test level begin during the corresponding development activity.
  • The tester should be involved in reviewing documents as soon as drafts are available in the development life cycle.
  • The V-model illustrates the testing aspects of verification and validation.

03 — Verification vs Validation

Verification vs Validation

Verification

Are we building the product right? To ensure that work products meet their specified requirements.

Validation

Are we building the right product? To ensure that the product actually meets the user's needs, and that the specifications were correct in the first place.

04 — QA vs QC

Quality Assurance vs Quality Control

Quality Assurance (QA)

A process that focuses on providing assurance that the quality request will be achieved. Aims to prevent causes of defect and correct them early in the development process. QA involves the full software development life cycle.

Quality Control (QC)

A process that focuses on fulfilling the quality requested. Aims to identify and fix defects. QC involves the full software testing life cycle. QC activities are only a part of the total range of QA activities.

05 — Test Levels

Test Levels

A test level (test stage) is a group of test activities that are organised and managed together; a test level is linked to the responsibilities in a project. The test levels are:

  • Component Testing
  • Integration Testing
  • System Testing
  • Acceptance Testing
  • Regression Testing

06 — Component Testing

Component Testing

Also known as unit testing, module testing, or program testing. It is the testing of individual program units — such as procedures, functions, methods, or classes — in isolation.

Goal

The goal of unit testing is to ensure that the code written for the unit meets its specification, prior to its integration with other units. It tests both functionality and non-functional characteristics.

  • Component testing is (usually) performed by the developer.
  • In test-driven development (TDD), which relates to the Agile model, unit tests are prepared by developers before the code was written.
  • As unit testing is made in isolation from the rest of the system, there can be missing parts of software.

Stubs and Drivers

Stubs and drivers replace the missing software and simulate the interface between components in a simple manner. A stub is called from the component to be tested; a driver calls a component to be tested.

StubsDrivers
Used in the top-down testing approach.Used in the bottom-up testing approach.
Used when the major module is ready to test, but the sub-modules are not ready yet.Used when the sub-modules are ready, but the main module is not ready yet.
Stubs are "called" programs.Drivers are "calling" programs.

07 — Integration Testing

Integration Testing

After units are written, putting them together to create the system is called integration — building something large from smaller pieces. Integration testing tests interfaces between components, interactions to different parts of a system (operating system, file system and hardware) or interfaces between systems.

Component integration testing

Verifies the interactions between software components and is done after component testing. Usually carried out by developers.

System integration testing

Verifies the interactions between different systems and may be done after system testing each individual system. Usually carried out by testers.

There are three commonly used integration strategies: Big-bang integration and Incremental integration (which is further divided into top-down and bottom-up).

Big-Bang Integration

All components are integrated simultaneously, then everything is tested as a whole. Individual modules are not integrated until all the modules are ready. In theory, combining already-tested components at once seems to save time — but this rests on the false assumption of no faults.

Advantage

Everything is finished before integration testing starts.

Disadvantages

Very time consuming; very difficult to trace the cause of failures because of the late integration; higher chance of critical failures from integrating everything at once; hard to detach modules to find the root cause of a bug; high probability of critical bugs in the production environment.

Incremental Integration

Builds up tested baselines: Baseline 0 = a tested component; Baseline 1 = two components; Baseline 2 = three components, etc.

  • Easier fault location and fix.
  • Easier recovery from disaster / problems.
  • Interfaces should have been tested in component tests.
  • You simply add to a tested baseline.

Top-Down Integration

Baselines grow from the top: baseline 0 = a; baseline 1 = a + b; baseline 2 = a + b + c; baseline 3 = a + b + c + d; etc. It must call lower-level components not yet integrated, so stubs simulate the missing components.

Advantages

Critical control structure tested first and most often; can demonstrate the system early (show working menus).

Disadvantages

Needs stubs; detail left until last; may be difficult to "see" detailed output (should have been tested in component test); may look more finished than it is.

Bottom-Up Integration

Baselines grow from the bottom: baseline 0 = n; baseline 1 = n + i; baseline 2 = n + i + o; baseline 3 = n + i + o + d; etc. It needs drivers to call the baseline configuration, and also needs stubs for some baselines.

Advantages

Lowest levels tested first and most thoroughly (should have been tested in unit testing); good for testing interfaces to the external environment (hardware, network); visibility of detail.

Disadvantages

No working system until the last baseline; needs both drivers and stubs; major control problems found last.

Minimum Capability Integration (also called Functional)

Baselines: baseline 0 = a; baseline 1 = a + b; baseline 2 = a + b + d; baseline 3 = a + b + d + i; etc. It needs stubs and shouldn't need drivers (if top-down).

Advantages

Control level tested first and most often; visibility of detail; a real working partial system earliest.

Disadvantage

Needs stubs.

Thread Integration (also called Functional)

The order of processing some event (interrupt, user transaction) determines the integration order — it is "minimum capability in time".

Advantages

Critical processing first; early warning of performance problems.

Disadvantage

May need complex drivers and stubs.

Integration Guidelines

  • Minimise support software needed.
  • Integrate each component only once.
  • Each baseline should produce an easily verifiable result.
  • Integrate small numbers of components at once — one at a time for critical or fault-prone components; combine simple related components.

Integration Planning

Integration should be planned in the architectural design phase. The integration order then determines the build order: components are completed in time for their baseline, and component development and integration testing can be done in parallel, which saves time.

08 — System Testing

System Testing

Testing an integrated system to verify it meets specified requirements. May include tests based on risks and/or requirement specifications, business process, use cases, or other high-level descriptions of system behavior, interactions with the operating system, and system resources.

  • Carried out by professional testers or independent testers.
  • Non-functional tests include performance and reliability; testers may also need to deal with incomplete or undocumented requirements.
  • Functional requirements testing starts by using the most appropriate specification-based (black-box) techniques.
  • System testing requires a product environment.

09 — Acceptance Testing

Acceptance Testing

Software testing performed prior to delivery. Its goal is not to find defects, but to give end users confidence that the system will function according to their expectations and can be released. How much is needed depends on the product risk.

Four typical forms

  • Contract and regulation acceptance testing
  • User acceptance testing (UAT)
  • Operational acceptance testing (OAT)
  • Alpha and beta testing

Alpha testing

Testing by potential users or an independent test team at the developing organization's site, before the product is released to external customers. It is a form of internal acceptance testing.

Beta testing

Testing by a group of customers / potential users who use the product at their own locations and provide feedback at the customer's site, before release. It is a form of external acceptance testing.
Alpha and beta testing are usually performed on commercial off-the-shelf software (COTS) — software developed for the mass market — to get feedback before release. Alpha testing is done before beta testing.

10 — Test Types

Test Types

A test type is a group of test activities aimed at testing a component or system focused on a specific test objective. The test types are:

  • Functional testing
  • Non-functional testing
  • Structural testing
  • Testing related to changes (re-testing and regression testing)

11 — Functional Testing

Functional Testing

Testing of functions — testing what the system does. Its objective is to test each function by providing appropriate input and verifying the output against the functional requirements. Also called specification-based testing.

  • The functionality is usually described in functional specifications, requirements specifications, business requirements, use cases and user stories.
  • Functional testing mainly involves black box testing and is not concerned about the source code of the application.
  • It focuses on: Security, Suitability, Accuracy, Interoperability (compatibility) and Compliance.

12 — Non-Functional Testing

Non-Functional Testing

Testing of non-functional software characteristics — testing of how the system works. Targets characteristics that do not relate to functionality: Reliability, Usability, Efficiency, Maintainability and Portability.

Performance testing

Performance testing determines how efficiently a product handles a variety of events (e.g. measuring a website's response time). The tester checks the product's speed (response is quick), scalability (can handle the maximum user load) and stability (stable under varying load). Sub-types:

Sub-typeDefinition (per slides)
Load testingEvaluates the performance of the system under the expected workload to determine what load can be handled (e.g. numbers of parallel users, numbers of transactions).
Stress testingEvaluates a system or component at or beyond the limits of its anticipated or specified workloads, or with reduced availability of resources such as memory or servers.
Endurance testingChecks system performance under specific load conditions over an extended / longer amount of time. Also known as soak testing (reveals issues like memory leaks).
Volume testingTests a software application or product with a certain amount of data (e.g. expanding a database to a specific size, then testing performance on it).
Spike testingSubjects the system to a short burst of concurrent load (e.g. an auction or e-commerce site where users increase suddenly in a very short duration).

Other non-functional test types

  • Document testing — verifies the technical accuracy and readability of user manuals, tutorials and online help (spelling and grammar checks, manual review to remove ambiguity or inconsistency). Can start at the very beginning of the software process, saving large amounts of money.
  • Installation testing — checks that the software has been correctly installed with all inherent features and works as expected; also known as implementation testing; done in the last phase of testing before the end user's first interaction.
  • Reliability testing — ensures the software is reliable, satisfies its purpose for a specified amount of time in a given environment, and is capable of rendering a fault-free operation.
  • Security testing — determines if an information system protects data and maintains functionality as intended (e.g. passwords stored encrypted, invalid users blocked, checking cookies and session time).

13 — Structural Testing

Structural Testing

Testing of the structure of the system or component — testing through assessment of coverage of a type of structure. Often referred to as 'white box' testing, because we are interested in what is happening inside the system/application.

  • Can be used at all levels of testing.
  • Developers use structural testing in component testing and component integration testing, especially where there is good tool support for code coverage.
  • Coverage measurement tools assess the percentage of executable elements (e.g. statements or decision outcomes) that have been exercised (i.e. covered) by a test suite.

14 — Testing Related to Changes

Testing Related to Changes

Objective: test changes related to fixed defects and look for unintended changes. When a defect has been corrected, two types of tests should be executed: confirmation testing (re-testing) and regression testing.

Confirmation testing (re-testing)

Testing a defect again after it has been fixed. When developers have fixed the defect, it is assigned for testers to re-test the test cases with the same inputs, data and environment to confirm the defect was really fixed.

Regression testing

Testing of a previously tested program following modification, to ensure that defects have not been introduced or uncovered in unchanged areas of the software as a result of the changes made. New defects can appear when the software was changed (adding/fixing functionality, refactoring) or when the environment was changed.

15 — Takeaways

Summary

For every development activity there is a corresponding testing activity — the V-Model pairs each specification/construction level with its own test level.
  • Verification = building the product right; validation = building the right product.
  • QA = prevention, across the whole development life cycle; QC = detection and fixing, across the testing life cycle.
  • Test levels: component → integration (big-bang or incremental: top-down, bottom-up, minimum capability and thread) → system → acceptance.
  • Test types: functional, non-functional, structural, and testing related to changes.
