White-Box Testing

White-box test techniques look inside the program and exercise its internal logic. Tests are derived from the code's structure — its statements, branches, conditions and loops — using a control flow graph, coverage criteria, cyclomatic complexity and basis path testing to decide which paths and test data are needed.

01 — Coverage Criteria

Basic Coverage Criteria

White-box testing measures how thoroughly the code's structure is exercised. The basic coverage criteria are:

  • Function coverage — has each function (or subroutine) in the program been called?
  • Statement coverage — has each statement in the program been executed?
  • Branch / Decision coverage — has each branch (also called a DD-path) of each control structure (such as in if and case statements) been executed? Given an if, have both the true and false branches been executed? Another way of saying this: has every edge in the program been executed?
  • Condition coverage (or predicate coverage) — has each Boolean sub-expression evaluated both to true and false?
  • Basic-Path coverage.

Consider the C function foo(int x, int y) whose body is if ((x > 0) && (y > 0)) z = x; The slides illustrate each criterion with this example:

CriterionSatisfying testsWhy
Function coveragefoo called at least onceThe function is invoked.
Statement coveragefoo(1,1)Every line — including z = x; — is executed.
Branch coveragefoo(1,1) and foo(0,1)First runs the true branch (z = x;); second fails (x>0) so the assignment is skipped.
Condition coveragefoo(1,0) and foo(0,1)Each Boolean sub-expression (x>0) and (y>0) is made both true and false.
Branch coverage subsumes statement coverage; condition coverage forces each Boolean sub-expression to take both values.

02 — The CFG

Control Flow Graph (CFG)

A Control Flow Graph models the possible flow of control through a program. It has three kinds of nodes and directed edges:

  • Statement nodes — represent single-entry-single-exit sequences of statements;
  • Predicate nodes — represent conditions for branching;
  • Auxiliary nodes — optional, for easier understanding (e.g. "join points" for an IF).

Edges represent the possible flow of control. It is relatively easy to map standard programming-language constructs to elements of a CFG: a sequence becomes one node arrowing down to the next; a branch/merge (an IF) becomes a predicate node that splits to a then-node and an else-node which rejoin at a join point; and a loop becomes a predicate node whose body loops back up to the condition.

Constructing a CFG — example

For the while/if-else snippet (slide 292), the statements/predicates map to numbered nodes 1–8 connected top to bottom. The while condition is node 2; the if condition is node 4, which branches to the then-branch (node 5) and the else-branch (node 6); these rejoin at node 7; back-edges loop the lower nodes up to node 2; node 8 is the exit node.

Paths and the CFG expression

A path is a route through the graph from entry to exit. A CFG expression writes the set of paths algebraically. For the nested if-else with entry node n₀ and exit node nₑ (slide 293/294), where node 1 is a==0, node 2 is b==0, node 3 is x=-b/a, node 4 is "Countless solutions" and node 5 is "No solution":

G1 = n₀ 1 2 4 nₑ + n₀ 1 2 5 nₑ + n₀ 1 3 nₑ
G1 = n₀ 1 (2 4 + 2 5 + 3) nₑ
G1 = n₀ 1 (2 (4 + 5) + 3) nₑ

03 — Nodes

Node-Coverage / Statement-Coverage Testing

For the snippet if(x==0) x++; g=1/x; the control flow graph has entry n₀ → node 1 (the predicate x==0). From node 1 the x=0 edge goes to node 2 (x++) and the x!=0 edge goes to node 3 (g=1/x); node 2 also flows to node 3, then node 3 → exit nₑ.

To cover all nodes we need the path that visits node 2, i.e. test data with x==0 following n₀ 1 2 3 nₑ. Comment from the slides: covering all nodes (all statements) does not necessarily exercise both branches — node coverage is the weakest of the criteria.

04 — Branches

Branch / Decision-Coverage Testing

For if(a < 2 && a==b) x = 2-a; else x = a-2; the CFG has n₀ → node 1 (the decision). The branch a<2 và a=b (true) goes to node 3 (x=2-a); the branch a≥2 hoặc b≠a (false) goes to node 2 (x=a-2); both converge at exit nₑ.

Branch coverage requires test data that drives the decision to true and to false — every edge of the decision must be taken at least once.

05 — Conditions

Condition-Coverage Testing

Condition coverage requires each Boolean sub-expression in a compound decision to evaluate both to true and to false. Because a decision such as a<2 && a==b is built from two atomic conditions, the CFG splits each predicate (a<2 ? then a=b ?) into separate nodes, each with its own T and F edges leading toward the x=2-a or x=a-2 outcomes.

The triangle classifier (slide 301/302) shows a chain of conditions: each sub-expression of (a+b<=c) || (b+c<=a) || (a+c<=b) is a decision node whose T branch goes to the Error node and whose F branch continues to the next condition, before the later Equilateral / Isoceles / Scalene tests. The IsInValidDate example (slide 303/304) chains year==1582, month==10, day>=5 and day<=14: each true edge cascades to the next condition and finally to return True, while any false edge converges to the return False node.

06 — Loops

Loop Testing

Loop testing exercises the iteration constructs of a program. In the CFG a loop appears as a predicate node (the loop condition) with a body that loops back to the condition and an exit edge when the condition fails. For the while(n<=m) example (slide 306/307), node a is the loop condition: the n≤m edge enters the body node b (i=n; s=s+a[i]; n++) which loops back to a, while the n>m edge exits toward node e (1/s) and on to nₑ. The IsThirtyDayMonth for-loop (slide 308) is the same idea applied to a counted loop that scans {4,6,9,11}.

07 — Complexity

Cyclomatic Complexity

Definition

The cyclomatic complexity of the flow chart indicates the number of independent paths of the program.

It can be computed three equivalent ways:

1.  V(G) = Number of closed regions (including the surrounding rectangle) + 1
2.  V(G) = Number of binary predicates + 1
3.  V(G) = E − N + 2      (Edges − Nodes + 2)

All three methods give the same value for the same flow graph.

08 — Basis Paths

Basis Path Testing

Basis path testing derives a set of independent paths from the program's decision logic. Its steps are:

  1. Construct a flow graph based on the program's decision logic.
  2. Compute the cyclomatic complexity of the flow graph.
  3. Determine the basis paths.
  4. Design test cases to go through these basis paths.

Baseline Path Method

A simple approach to identify basis paths:

  1. Find the shortest path from entry to exit.
  2. Return to the function entry point.
  3. Change one conditional expression to its alternate outcome.
  4. Follow the shortest path to the exit point.
  5. Repeat until all basis paths are defined.
  6. Identify data to execute the paths.

09 — Worked Example

White-Box Testing Example — FindMean

The FindMean function reads scores from a file, sums the positive ones, then computes and prints the mean (or reports that no scores were found). It is taken through the full method.

Prepare the flow graph

The code is partitioned into nine flow-graph nodes:

NodeCode it represents
1Initialization block (declarations and the first Read)
2while (! EOF(ScoreFile)) — loop condition
3if (Score > 0.0) — condition
4Loop body: SumOfScores += Score; NumberOfScores++;
5Read(ScoreFile, Score); inside the loop
6if (NumberOfScores > 0) — condition
7True branch: compute and print the mean
8Else branch: print "No scores found in file"
9Exit / final node

Construct the CFG

Node 1 flows to node 2 (the while). The loop region contains the inner if (node 3) which branches to node 4 (then) and skips to node 5 (else), with node 5 looping back to node 2. On exit, node 6 (the final if) branches in a diamond to node 7 and node 8, which reconverge at the exit node 9.

Cyclomatic complexity

All three formulas agree on the same value for this graph:

V(G) = 4 (closed regions)                 = 4
V(G) = 3 (conditional nodes) + 1          = 4
V(G) = E − N + 2 = 11 − 9 + 2             = 4

Basis path / paths analysis

Because of the dependency "if node 4 is included then node 7 must be included" (a positive score must have been counted), some path shapes are not feasible:

PathFeasible?Test data
1, 2, 6, 8, 9okNo records in the file
1, 2, 6, 7, 9not possible—
1, 2, 3, 4, 5, 2, 6, 7, 9okOne positive score
1, 2, 3, 4, 5, 2, 6, 8, 9not possible—
1, 2, 3, 5, 2, 6, 8, 9okOne negative score
1, 2, 3, 5, 2, 6, 7, 9not possible—
These 3 feasible tests cover all paths — we did not need 4! Every line of code is tested.

10 — Takeaways

Summary

White-box testing derives tests from the code's structure: build the CFG, choose a coverage criterion, compute cyclomatic complexity, and use basis paths to pick the test data.

Function, statement, branch, condition and basic-path coverage form a ladder of increasing thoroughness. The control flow graph turns code into nodes and edges; cyclomatic complexity V(G)=E−N+2 counts the independent paths; and basis path testing with the baseline path method selects exactly those paths — feasibility permitting — so that every line is exercised with the fewest tests.
