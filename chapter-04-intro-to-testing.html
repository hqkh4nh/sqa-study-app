Introduction to Software Testing

Software quality is the degree to which a system, component, or process meets specified requirements. This chapter reviews the IEEE definition of quality; the distinction between error, fault and failure; the causes and high cost of software defects; the definitions of software testing; the 7 principles of testing; why testing is necessary and its objectives; and the five stages of the fundamental test process.

01 — Quality

What is Software Quality?

Definition

Software quality — IEEE definition: the degree to which a system, component, or process meets specified requirements.

Quality is ultimately reflected in customer satisfaction, which the slides rate on a scale from Very Poor and Poor, through Average, up to Good and Excellent.

02 — Error · Fault · Failure

Error? Fault? Failure?

These three terms are distinct and must not be used interchangeably:

Definition

Error — also known as a mistake. A software error is made by human action which produces an incorrect result. An error can be a Syntax (grammatical) error or a Logic error.

Definition

Fault — also known as a defect or bug. A fault is a manifestation of an error in software. All software errors may not cause software faults.

Definition

Failure — a fault becomes a failure if it is activated / executed. A failure is a deviation of the software from its expected delivery or service. Not all faults result in failures; some stay dormant in the code and we may never notice them.

Failure is an event; Fault is a state of the software, caused by an error.

03 — The Chain

A Person Makes an Error

The slides illustrate a three-step chain of causation:

  1. A person makes an error …
  2. … that creates a fault in the software …
  3. … that can cause a failure in operation.

Viewed across the software development process, errors lead to faults, and some of those faults propagate down to failures — but not all of them do.

04 — Causes

Causes of Software Errors

Errors may occur for many reasons, such as:

  • Time pressure;
  • Human is error prone;
  • Inexperienced or insufficiently skilled project participants;
  • Miscommunication between project participants, including miscommunication about requirements and design;
  • Complexity of the code, design, architecture, the underlying problem to be solved, and the technologies used;
  • New, unfamiliar technologies.

The 5 Causes of Software Errors

The slides' mind map groups the causes into five categories:

#Cause
1Inadequate / poorly specified system or project requirements
2Logical design errors (e.g. erroneous algorithm, erroneous definition of boundary conditions)
3Coding errors — too many to list (syntax, logic, run-time errors)
4Non-compliance with documentation and coding procedures
5Deliberate deviation from software requirements (e.g. reuse, budget / schedule pressure)

05 — The Cost

High Cost of Software Defects

Testing is the way to remove defects, and the economic stakes are enormous:

  • $59.5 billion / year in the USA — "The Economic Impacts of Inadequate Infrastructure for Software Testing" report for the National Institute of Standards & Technology (NIST), US Department of Commerce, 2002.
  • Research by CISQ (Consortium for Information & Software Quality) found that, in 2018, poor quality software cost organizations $2.8 trillion in the US alone.
  • On average, software developers make 100 to 150 errors for every thousand lines of code.

06 — Definitions

Definitions of Software Testing

There is a variety of definitions for software testing:

  • Testing is the process of executing a program with intention of finding errors (Myers).
  • The purpose of testing is to find errors; testing is the process of trying to discover every conceivable fault or weakness in a work product (Myers, Kit).
  • The process of searching for errors (Kaner).
  • The process of finding errors and of validating the program / system (Jorgensen).
  • The purpose of software testing is to find errors and to get them fixed (Bitzenhofer).
  • The purpose of software testing is to reduce risk (E&M).

The Formal Definition

Software testing is a formal process carried out by a specialized testing team in which a software unit, several integrated software units or an entire software package are examined by running the programs on a computer. All associated tests are performed according to approved test procedures on approved test cases.

KeywordMeaning
FormalScheduled in advance; central in the development process; not ad hoc.
Specialized Testing TeamIndependent test team or external consultancy. Unit tests by developers — okay, but that is it.
Running the programA must; nothing static.
Approved Test ProceduresUse a plan that is approved.
Approved Test CasesAll planned; all tests undertaken.
Definition

Software testing is the process consisting of all lifecycle activities, both static and dynamic, concerned with planning, preparation and evaluation of software products and related work products to determine that they satisfy specified requirements, to demonstrate that they are fit for purpose, and to detect defects.

07 — The 7 Principles

Principles of Testing

There are seven principles of testing:

#PrincipleWhat it means
1Testing shows the presence of defectsTesting can show that defects are present, but cannot prove that there are no defects.
2Exhaustive testing is impossibleTesting everything — all preconditions and combinations of inputs — is impossible; testers should apply risk analysis and set priorities.
3Early testingTesting activities should start as early as possible in the life cycle and be focused on defined objectives.
4Defects clusteringA small number of modules contain most of the defects discovered during pre-release testing or show the most operational failures.
5Pesticide paradoxRepeating the same tests eventually finds no new bugs; test cases must be regularly reviewed and revised, with new and different tests written.
6Testing is context dependentTesting is done differently in different contexts — e.g. safety-critical software is tested differently from an e-commerce site.
7Absence of errors fallacyFinding and fixing defects does not help if the system is unusable and does not fulfill the users' needs and expectations.

Why Exhaustive Testing is Impossible

08 — Why & What For

Why Testing is Necessary & Objectives

Why Testing is Necessary

  • All software has defects (bugs);
  • Software products are getting larger and more complicated;
  • Software is written by people — people make mistakes;
  • Some problems might be trivial, but others can be costly and damaging — loss of money, time, or business reputation — and may even result in injury or death;
  • Not all software systems carry the same level of risk, and not all problems have the same impact when they occur.

The cost of defects rises considerably across the life cycle — on a logarithmic scale, the cost of repair grows from roughly 1 at Specification, to 10 at Design, to 100 at Programming, to 1000 in Production. Testing therefore looks to find the most important defects as early as possible, increasing confidence that the software meets specification and helping to measure the quality of software.

Software Testing Objectives

  • To identify and reveal as many errors as possible;
  • To gain confidence about the level of quality;
  • To prevent defects;
  • To provide information for decision-making (stakeholders);
  • To reduce the level of risk of inadequate software quality;
  • To make sure the system works as expected and meets the user requirements;
  • To comply with contractual, legal or regulatory requirements or standards.

09 — The Process

The Fundamental Test Process

The fundamental test process has five stages, running in sequence with feedback loops back up and a Control activity running alongside throughout:

#Stage
1Test Planning (and Control)
2Analysis and Design
3Implementation and Execution
4Evaluating Exit Criteria and Reporting
5Test Closure Activities

1. Test Planning

"A goal without a plan is just a wish." Major tasks are: identify the objectives of testing; determine scope; determine the test approach; determine the required test resources; implement the test policy and/or the test strategy; schedule test analysis and design tasks; schedule test implementation, execution and evaluation; and determine the exit criteria.

Test Control is the ongoing activity of comparing actual progress against the plan, reporting status (including deviations from the plan), and taking actions necessary to meet the mission and objectives of the project. Major control tasks: measure and analyze results; monitor and document progress, test coverage and exit criteria; initiate corrective actions; and make decisions.

2. Analysis and Design

"Analysis gain better understanding." Review the Test Basis — evaluating the testability of the Test Basis and Test Object(s); identify and prioritize Test Conditions and associated Test Data (documented in a Test Design Specification); design and prioritize the Test Cases; identify required test data; design the test environment set-up; and identify any required infrastructure and tools.

3. Implementation and Execution

"Ideas are easy. Implementation is hard." Develop, implement and prioritize test cases; create the Test Scripts; create test data; write automated test scripts; and check the environment — verify that the test environment has been set up correctly.

4. Evaluating Exit Criteria and Reporting

"Evaluation — learning tool to improve." Evaluating exit criteria is the activity where test execution is assessed against the defined objectives. Exit criteria should be set and evaluated for each test level; check test logs against the exit criteria specified in the test planning; assess if more tests are needed or if the exit criteria should be changed; and write a test summary report to stakeholders.

How to measure exit criteria? All planned requirements must be met; all high priority bugs should be closed; all test cases should be executed; if the scheduled time out is arrived; and the test manager must sign off the release. Note: all these parameters can be met by percentages (not 100%).

5. Test Closure Activities

"Closure — way to start a new beginning." Collect data from the complete test activity; finalize and archive the test ware (scripts, test environment, etc.); and evaluate how testing went, analyzing lessons learned for future releases and projects.

10 — Takeaways

Summary

Testing shows the presence of defects, but can never prove their absence — so start early, prioritize by risk, and follow a planned, formal process.

Quality is meeting specified requirements. A person makes an error that creates a fault that can cause a failure, though not all faults fail. Because defects are costly, testing — a formal, lifecycle-wide process guided by seven principles — exists to reveal errors, build confidence, and reduce risk, delivered through the five-stage fundamental test process.
